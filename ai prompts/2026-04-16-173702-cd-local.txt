● I found the bug. Here's exactly what's happening:

  Root cause: Race condition between two async operations on page load.

  1. main.js (line 235) calls CosmeticStore.load().then(...) which calls
  renderGrid() when cosmetics load
  2. renderGrid() calls getCurrentXp() (main.js:44) which reads
  window._sbXp.totalXp
  3. But window._sbXp isn't fully initialized yet — _totalXp starts at 0 in the
  constructor and only gets set to the real value after await 
  window._sbXp.init() finishes
  4. So if cosmetics load before the XP init completes, getCurrentXp() returns 0
   → all items render as locked

  The tab-switch clears it because after returning to the tab, the user re-opens
   the customize panel — by then window._sbXp.init() has long finished, but the
  grid was never re-rendered with the correct XP.

  The fix: after window._sbXp.init() completes in index.html, call renderGrid()
  again.

> and will this fix it?                                                         

● Yes, it should fix the "stays locked until you switch tabs" part. Here's why:

  Before the fix:
  1. Page loads → renderGrid() fires early (before XP loads) → items show as
  locked
  2. XP finishes loading → nothing triggers another renderGrid() → items stay
  locked forever until something else triggers a re-render (like returning to
  the tab from a navigation)

  After the fix:
  1. Page loads → renderGrid() fires early → items show as locked (briefly)
  2. XP finishes loading → renderGrid() fires again with the real XP → items
  show as unlocked automatically

  There may still be a very brief flash where items look locked right as the
  page loads, but they'll correct themselves immediately without needing to
  switch tabs. In practice, users won't notice because they have to navigate to
  the customize panel first, by which time the XP will have already loaded.
