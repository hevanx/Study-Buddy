  A) Schema Analysis

  The existing friendships table:

  friendships
    id           INT UNSIGNED PK AUTO_INCREMENT
    sender_id    INT UNSIGNED  → FK users.id CASCADE DELETE
    receiver_id  INT UNSIGNED  → FK users.id CASCADE DELETE
    status       ENUM('pending','accepted','declined')  DEFAULT 'pending'
    created_at   DATETIME  DEFAULT current_timestamp()
    updated_at   DATETIME  ON UPDATE current_timestamp()
    UNIQUE KEY uq_friendship (sender_id, receiver_id)
    KEY fk_friendship_receiver (receiver_id)          ← already indexed ✓

  Assessment — all requirements are met with one behavioural fix:
  The declined status creates a dead-end row that blocks re-sending. Fix: on 
  decline, delete the row rather than setting status='declined'. This keeps the
  table clean and lets the sender try again later. The declined enum value can
  stay — we just won't use it.

  No ALTER TABLE is needed. The schema is complete.
